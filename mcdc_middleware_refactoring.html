01 The Problem

A real middleware function where decisions are tangled with side effects

This is createMiddleware from a Go API gateway. It's ~80 lines of HTTP handler logic with 6 boolean decisions interleaved with tracing, logging, metrics, and error handling:

Go — Original (problems highlighted in comments) 
return func(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // DECISION 1: Should we trace? (tangled with span creation)
        if baseMw := mw.Base(); baseMw != nil {
            cfg := baseMw.Gw.GetConfig()
            if cfg.OpenTelemetry.Enabled && baseMw.Spec.DetailedTracing {
                ctx, span := baseMw.Gw.TracerProvider.Tracer().Start(r.Context(), mw.Name())
                setContext(r, ctx)
                defer func() {
                    attrs := ctxGetSpanAttributes(r, mw.Name())
                    if len(attrs) > 0 { span.SetAttributes(attrs...) }
                    span.End()
                }()
            }
        }

        logger := mw.Base().SetRequestLogger(r)

        // DECISION 2: NewRelic segment (tangled with defer)
        if txn := newrelic.FromContext(r.Context()); txn != nil {
            defer txn.StartSegment(mw.Name()).End()
        }

        // DECISION 3: Instrumentation (repeated 3x throughout function)
        if instrumentationEnabled {
            meta = health.Kvs{
                "from_ip":  request.RealIP(r),
                "method":   r.Method,
                "endpoint": r.URL.Path,
                // ... more fields ...
            }
        }

        // DECISION 4: CORS passthrough (early return, skips everything)
        if mw.Base().Spec.CORS.OptionsPassthrough && r.Method == "OPTIONS" {
            next.ServeHTTP(w, r)
            return
        }

        err, errCode := mw.ProcessRequest(w, r, mwConf)

        // DECISION 5: Error handling + GoPlugin exception (nested)
        if err != nil {
            writeResponse := true
            if goPlugin, isGoPlugin := actualMW.(*GoPluginMiddleware); isGoPlugin && goPlugin.handler != nil {
                writeResponse = false  // GoPlugin already wrote the response
            }
            handler := ErrorHandler{mw.Base()}
            handler.HandleError(w, r, err.Error(), errCode, writeResponse)
            // ... metrics, logging ...
            return
        }

        // ... metrics, logging ...
        mw.Base().UpdateRequestSession(r)

        // DECISION 6: StatusRespond bypass
        if errCode != middleware.StatusRespond {
            next.ServeHTTP(w, r)
        }
    })
}

Go — Pure Decision Functions 
// PlanBeforeProcess: everything knowable before ProcessRequest.
// Pure function. No side effects. No interfaces. Deterministic.
func PlanBeforeProcess(in MiddlewareInputs) MiddlewarePlan {
    plan := MiddlewarePlan{
        Instrument: in.InstrumentationOn,
        Trace:      in.OTelEnabled && in.DetailedTracing,  // Decision 1
    }

    plan.PassthroughCORS = in.CORSPassthrough &&
        in.RequestMethod == "OPTIONS"                       // Decision 4
    plan.Process = !plan.PassthroughCORS

    if plan.PassthroughCORS {
        plan.CallNext = true
        plan.WriteError = false
        plan.UpdateSession = false
    }

    return plan
}

// PlanAfterProcess: what to do with the ProcessRequest result.
// Pure function. Only called when plan.Process == true.
func PlanAfterProcess(plan MiddlewarePlan, result ProcessResult) MiddlewarePlan {
    hasError := result.Err != nil

    if hasError {
        // Decision 5: GoPlugin suppresses error response write
        plan.WriteError = !(result.IsGoPlugin && result.PluginHasHandler)
        plan.CallNext = false
        plan.UpdateSession = false
    } else {
        plan.WriteError = false
        // Decision 6: StatusRespond bypass
        plan.CallNext = result.ErrCode != middleware.StatusRespond
        plan.UpdateSession = true
    }

    return plan
}

The Orchestrator — Zero Decision Logic

Go — Orchestrator 
// Inside the HTTP handler closure. No decisions — just wiring.
inputs := MiddlewareInputs{
    OTelEnabled:       cfg.OpenTelemetry.Enabled,
    DetailedTracing:   baseMw.Spec.DetailedTracing,
    CORSPassthrough:   baseMw.Spec.CORS.OptionsPassthrough,

// DECIDE (pure, tested by FRET + MC/DC)
plan := PlanBeforeProcess(inputs)
exec := NewExecutor(mw, mwConf, r)

// EXECUTE (mechanical — each line is one plan field, one action)
if plan.Trace      { exec.SetupTracing(r); defer exec.FinalizeTracing(r) }
if plan.Instrument { meta = exec.RecordMetrics(r) }
if plan.PassthroughCORS { next.ServeHTTP(w, r); return }

// Process request
procErr, errCode := mw.ProcessRequest(w, r, mwConf)

// DECIDE again (pure, tested by FRET + MC/DC)
plan = PlanAfterProcess(plan, ProcessResult{
    Err: procErr, ErrCode: errCode,
    IsGoPlugin:       isGoPluginMiddleware(actualMW),
    PluginHasHandler: goPluginHasHandler(actualMW),
})

// EXECUTE results (mechanical — no branching logic)
if plan.WriteError    { exec.HandleError(w, r, procErr, errCode, true) }
if plan.UpdateSession { mw.Base().UpdateRequestSession(r) }
if plan.CallNext      { next.ServeHTTP(w, r) }

func TestPlanBeforeProcess(t *testing.T) {
    tests := []struct {
        name   string
        in     MiddlewareInputs
        expect MiddlewarePlan
    }{
        // MC/DC for Trace: OTelEnabled && DetailedTracing (3 tests)
        {"trace: both enabled",
            MiddlewareInputs{OTelEnabled: true, DetailedTracing: true},
            MiddlewarePlan{Trace: true, Process: true}},
        {"trace: otel disabled",
            MiddlewareInputs{OTelEnabled: false, DetailedTracing: true},
            MiddlewarePlan{Trace: false, Process: true}},
        {"trace: tracing disabled",
            MiddlewareInputs{OTelEnabled: true, DetailedTracing: false},
            MiddlewarePlan{Trace: false, Process: true}},

        // MC/DC for CORS: CORSPassthrough && Method==OPTIONS (3 tests)
        {"cors: passthrough + OPTIONS",
            MiddlewareInputs{CORSPassthrough: true, RequestMethod: "OPTIONS"},
            MiddlewarePlan{PassthroughCORS: true, CallNext: true}},
        {"cors: passthrough + GET",
            MiddlewareInputs{CORSPassthrough: true, RequestMethod: "GET"},
            MiddlewarePlan{Process: true}},
        {"cors: no passthrough + OPTIONS",
            MiddlewareInputs{RequestMethod: "OPTIONS"},
            MiddlewarePlan{Process: true}},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := PlanBeforeProcess(tt.in)
            assert.Equal(t, tt.expect, got)
        })

}

func TestPlanAfterProcess(t *testing.T) {
    base := MiddlewarePlan{Process: true, Instrument: true}

    tests := []struct {
        name   string
        result ProcessResult
        expect MiddlewarePlan
    }{
        // R6: happy path
        {"no error, normal code",
            ProcessResult{ErrCode: 200},
            MiddlewarePlan{Process: true, Instrument: true,
                CallNext: true, UpdateSession: true}},

        // R5: StatusRespond bypass
        {"no error, StatusRespond",
            ProcessResult{ErrCode: middleware.StatusRespond},
            MiddlewarePlan{Process: true, Instrument: true,
                CallNext: false, UpdateSession: true}},

        // R2 + R3: error, not go plugin
        {"error, not go plugin",
            ProcessResult{Err: errors.New("fail"), ErrCode: 500},
            MiddlewarePlan{Process: true, Instrument: true,
                WriteError: true}},

        // R4: GoPlugin suppresses response write
        {"error, go plugin with handler",
            ProcessResult{Err: errors.New("fail"), ErrCode: 500,
                IsGoPlugin: true, PluginHasHandler: true},
            MiddlewarePlan{Process: true, Instrument: true}},

        // MC/DC: PluginHasHandler independence pair
        {"error, go plugin WITHOUT handler",
            ProcessResult{Err: errors.New("fail"), ErrCode: 500,
                IsGoPlugin: true, PluginHasHandler: false},
            MiddlewarePlan{Process: true, Instrument: true,
                WriteError: true}},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := PlanAfterProcess(base, tt.result)
            assert.Equal(t, tt.expect, got)
        })
    }
}

How requirements flow: SRS to FRET

NASA doesn’t use Jira tickets. They use formal Software Requirements Specifications (SRS) — documents where every requirement has a unique ID, traceable to design, code, and tests.

Traditional SRS (ambiguous)

“The system shall enter safe mode upon detection of anomalous sensor readings.”

  • What’s “anomalous”?
  • How fast? Immediately? Within 100ms?
  • Which sensor? All of them?
  • Developer interprets, asks questions, makes assumptions
FRET requirement (precise)

“When sensor_A_value > threshold, the FlightController shall immediately satisfy mode = SAFE_MODE.”

  • Which variable, what condition, what timing, what output — all explicit
  • Developer implements directly — inputs and outputs are named
  • Tester derives MC/DC cases directly from the conditions
  • Kind 2 proves it formally. Copilot monitors it at runtime.
ℹ
FRET augments the SRS, it doesn’t replace it. The SRS remains the authoritative document (hundreds of pages: functional, interface, safety, design constraints, quality attributes). Only the behavioral “shall” statements with clear input/output/timing get formalized in FRET (~40-60% of functional requirements, most safety requirements). Design constraints (“shall use C99”), quality attributes (“shall be maintainable”), and interface specs stay as traditional SRS text.
★
The “questions back” loop — arguably the most valuable part. Translating SRS → FRETish forces the requirements engineer to resolve every ambiguity. “Anomalous” becomes a specific threshold. “Promptly” becomes a cycle count. “Any primary sensor” becomes a named variable. These questions go back to systems engineers, improving the SRS itself. Defects caught at requirements stage are orders of magnitude cheaper than defects found in testing.
★
One artifact, three consumers. The FRETish formalization simultaneously serves as: (1) developer implementation spec (alongside the SRS), (2) tester’s MC/DC test spec, (3) verifier’s formal contract. It doesn’t add work — it replaces the ambiguity-resolution step that already happens informally (in people’s heads) with a tool-supported version that produces machine-checkable artifacts as a byproduct.

Where does the SRS live? DOORS + FRET together

In practice, the SRS doesn’t live as a Word document — it lives in IBM Rational DOORS (Dynamic Object-Oriented Requirements System), the industry-standard requirements management tool. DOORS handles storage, versioning, traceability. FRET handles formalization. They are complementary:

ConcernDOORSFRET
StoresALL requirements — functional, interface, safety, design constraints, qualityOnly formalizable behavioral subset (~40-60% of functional, most safety)
FormatEnglish text + attributes + traceability linksFRETish + temporal logic + CoCoSpec contracts
AnalysisCoverage gaps (link counting), change impact, baselinesRealizability, vacuity, formal verification, trace diagrams
OutputReports, matrices, approval workflowsCoCoSpec contracts, Copilot specs, JSON export

The complete multi-step flow

1. Systems engineers write requirements in DOORS (single source of truth, unique IDs, version control)
2. Requirements engineers identify formalizable subset → translate to FRET, preserving DOORS IDs
3. FRET forces precision → questions go back to systems engineers → DOORS requirements improve
4. FRET exports CoCoSpec contracts → Kind 2 verifies Lustre models (design-time proof)
5. FRET exports JSON → Ogma → Copilot C99 runtime monitors
6. Traceability in DOORS updated: DOORS-REQ-0100 → FRET:FSW-REQ-0100 → Kind2:proof → test:TestSafeMode → code:line-247
7. Non-formalizable requirements stay in DOORS only (verification: Inspection/Analysis)
8. IV&V uses both: DOORS for traceability auditing, FRET/Kind 2 for formal verification
ℹ
For software teams adapting this: You probably don’t have DOORS. The mapping: DOORS → Jira/Linear with structured fields. SRS → well-structured tickets with explicit acceptance criteria. FRET → FRETish formalizations attached to tickets. Kind 2 → design-time verification (optional). Copilot → runtime assertions / observability. MC/DC → table-driven tests in CI with requirement IDs in test names. The requirement ID is the thread connecting everything.

How the SRS evolves: bugs, features, extensions

The SRS is a living document under configuration management, not a one-time artifact. You never create a new SRS — you update the existing one. DOORS tracks every change: who, when, why, previous text.

EventSRS change?FRET change?New document?

Most bugs trace to requirements ambiguity, not coding errors. “Exceeding the threshold” — does that mean > or >=? In English, it’s ambiguous. In FRETish, sensor_A > threshold is not. If the requirement had been formalized from the start, the off-by-one would have been caught during translation, not in testing.

Why do GN&C and flight software work so differently?

The fundamental difference is the nature of the problem, not a process choice:

GN&C = Continuous Math

“Given attitude sensor X and thruster capacity Y, compute thrust vector Z.”

  • Mathematically expressible — differential equations, transfer functions, PID controllers
  • Naturally dataflow — inputs → transforms → outputs, every cycle, no side effects
  • Same computation every clock tick — no branching, no event-driven state
  • Simulink/SCADE were designed for exactly this — block diagrams map 1:1 to the math

Analogy: a signal processing pipeline (audio effects chain).

Flight Software = Event-Driven State

“When sensor A reports fault AND we’re in cruise phase, switch to safe mode, reconfigure power bus, notify ground.”

  • State machines with complex transitions — not continuous math
  • Event-driven — responds to commands, faults, timers
  • Rich I/O — hardware buses, file systems, comms links
  • Edge cases everywhere — fault handling, recovery sequences, timing
  • Simulink is awkward for this — like writing a novel in a spreadsheet

Analogy: a web server with state machines — like our middleware.

ℹ
Our middleware example is flight software, not GN&C. Event-driven, state-dependent, lots of branching, rich I/O. That’s why we face the same challenges NASA flight software engineers face — hand-written code, spec/implementation gap, process-heavy mitigation. The common thread across both NASA teams isn’t Simulink — it’s the separation of requirements, implementation, and verification into different teams with formal tooling connecting them.

Can FRET flag scenarios you haven’t covered?

Yes — this is one of FRET’s most valuable capabilities:

DetectionWhat it catchesExample
Vacuity detection Requirements that are “vacuously true” — trivially satisfied because their trigger condition never actually fires in any scenario R4 says “when error AND goPlugin, suppress response” — but if no other requirement ever makes both true together, R4 is dead code. Looks covered, isn’t.
Completeness gaps Input combinations where no requirement specifies what should happen R1–R6 cover error and happy paths, but nothing says what happens when error=false && errCode=0 && newFlag=true. That’s a gap — undefined behavior.
Conflict detection Two requirements that demand contradictory outputs for the same input R3 says “when error, write_error=true” and R4 says “when error AND goPlugin, write_error=false” — FRET checks if these can coexist.
Diagnostic traces The specific input combination that triggers a gap or conflict “Conflict between R3 and R4 when error=true, isGoPlugin=true, pluginHasHandler=false”
⚠
FRET’s gap detection has limits. It catches obvious gaps through realizability analysis, but full exhaustive “are ALL 2N scenarios covered?” checking is Kind 2’s job. Think of it as two layers: FRET catches requirement-level gaps (missing rules, conflicts, dead requirements). Kind 2 catches implementation-level gaps (does the code handle every state?). Together they cover both.

Step 1: FRET generates contracts (not implementations)

Each FRETish requirement (R1–R7) is formalized into past-time metric Linear Temporal Logic. FRET exports this as CoCoSpec contracts — the (*@contract ... *) blocks with guarantee clauses. These are the rules the system must obey.

⚠
FRET generates only the contracts (assume/guarantee), not the implementation. The Lustre node body — the actual equations like trace = otel_enabled and detailed_tracing — is a separate artifact. Kind 2 checks whether the implementation actually satisfies the FRET-generated contracts. This is the whole point: one team writes requirements, another writes code, the tool checks they agree.
ℹ
Is the Lustre implementation always written by hand? No — it depends on the workflow:

In NASA/aerospace: The system model often comes from Simulink — MATLAB’s visual programming environment for control systems. Engineers draw block diagrams: inputs, outputs, logic gates, state machines, lookup tables connected by wires. For our middleware, you’d literally drag-and-drop an AND gate for otel_enabled && detailed_tracing and wire it to a trace output. Simulink → Lustre translation is mechanical because both have the same semantics: “every clock tick, compute all outputs from all inputs, no side effects.” FRET generates the contracts. Engineers may never see raw Lustre — it’s the formal intermediate representation (like assembly is to C).

In automotive/avionics (SCADE): SCADE is the same idea but commercial and DO-178C certified. Engineers draw block diagrams, SCADE generates verified Lustre + certified C code. Airbus fly-by-wire uses this. Again — no hand-written Lustre.

In our Go middleware example: We’re working backwards from existing code that was never designed in Simulink or SCADE. So here the Lustre translation is manual. But that’s the exception in safety-critical flows, not the rule. The normal path is: requirements in FRET → design in Simulink/SCADE → Lustre generated automatically → Kind 2 verifies.

What happens when code changes? How do we know the implementation matches the spec?

This is the hardest question in the entire pipeline. The answer depends on which part of the system you’re looking at — even within NASA:

FlowWhoCode written how?Spec ↔ implementation gap?

NASA flight software Flight software engineers Hand-written C/C++ (Mars rovers, Orion, JWST) Same gap as us — code and spec are separate artifacts

★
Even NASA doesn’t fully close this gap for hand-written code. Auto-generation from the spec (SCADE) is only used for GN&C control algorithms. Flight software — sequencing, fault management, comms — is hand-written C/C++, and the spec-to-code conformance gap exists there too. NASA mitigates it with process rigor, not tooling magic.

Runtime monitors (R2U2 on separate hardware) Copilot C99 monitors checking live traffic against FRET requirements
⚠
Be honest in presentations: The spec-to-implementation gap is real for any hand-written code — including NASA’s own flight software. The difference is that NASA mitigates it with six layers of process controls (independent review, MC/DC, traceability, IV&V, static analysis, runtime monitors). We can adopt the same layers. The gap doesn’t close fully, but each layer shrinks it. Even with the gap, you’re far ahead of “we have unit tests and hope for the best.”

     and err_code <> STATUS_RESPOND)

let
  write_error = has_error and
    not (is_go_plugin and plugin_has_handler);

  call_next = if has_error then false
    else err_code <> STATUS_RESPOND;

    else true;

Go — Same logic
func PlanAfterProcess(
    plan MiddlewarePlan,
    result ProcessResult,
) MiddlewarePlan {
    hasError := result.Err != nil

    if hasError {
        plan.WriteError =
            !(result.IsGoPlugin &&
              result.PluginHasHandler)
        plan.CallNext = false
        plan.UpdateSession = false
    } else {
        plan.WriteError = false
        plan.CallNext =
            result.ErrCode !=
            middleware.StatusRespond
        plan.UpdateSession = true
    }

    return plan
}

Runtime Monitoring: The Third Layer

⚠
Why do we need runtime monitors if Kind 2 already proved everything?
Kind 2 proves that the Lustre specification is correct — but your production Go code is a separate implementation. Bugs can creep in during translation: a typo, a missed edge case when updating code, or a drift between the spec and the implementation over time. Runtime monitors act as a safety net that continuously checks whether the actual running system still respects the same requirements that were proven at design time. Think of it as the difference between proving a bridge design is sound on paper vs. installing strain gauges on the actual bridge.

The same FRET requirements that drive Kind 2 proofs can also generate Copilot runtime monitors via Ogma. These compile to C99 constant-memory monitors that run alongside your Go service and fire alerts if any requirement is violated in production:

Haskell — Copilot Monitor (generated from FRET) 
-- Generated by Ogma from FRET JSON export
-- Monitors R2: "error blocks forwarding"

spec :: Spec
spec = do
  -- External variables (fed from Go middleware via CGo or sidecar)
  let processError = extern "process_error" :: Stream Bool
  let callNext     = extern "call_next"     :: Stream Bool

  -- R2: When process_error is true, call_next must be false
  trigger "r2_violation"
    (processError && callNext)  -- fires if error=true AND next=called
    []

  -- R6: When no error and not StatusRespond, call_next must be true
  let statusRespond = extern "err_code_is_respond" :: Stream Bool
  trigger "r6_violation"
    (not processError && not statusRespond && not callNext)
    []

Copilot compiles this to C99 with zero dynamic allocation, constant memory, constant time per step. The monitor observes every request decision and fires a trigger if any requirement is violated in production.

ℹ
But what validates the monitors themselves? The trust chain works like this:

1. You review the FRET requirements (R1–R7) — these are human-readable, auditable English sentences.
2. FRET mechanically translates them to temporal logic — no hand-written code, no room for translation error.
3. Ogma mechanically generates Copilot Haskell from the same temporal logic — again, no hand-written code.
4. Copilot's compiler is itself formally verified — it guarantees the generated C99 faithfully implements the Haskell spec.

The only trust assumption is that the FRET requirement is correct. Every step after that is mechanical or verified. This is the same assurance model NASA uses for flight-critical monitors.

09 The Key Insight

Same principle at every level

The Decide → Plan → Execute pattern is the FRET/Lustre component model applied at the function level:

  • Inputs (MiddlewareInputs, ProcessResult) = FRET Input variables = Lustre node inputs
  • Decision functions (PlanBeforeProcess, PlanAfterProcess) = FRET component = Lustre node body
  • Plan struct = FRET Output variables = Lustre node outputs
  • Contracts (R1–R7) = FRETish requirements = CoCoSpec guarantees = MC/DC test tables
  • Executor = outside the formal boundary — mechanical side effects, integration-tested

The pattern works because it separates what to do (pure, provable, monitorable) from how to do it (side effects, I/O). This is exactly what makes Lustre effective: synchronous dataflow equations describe behavior, compilation handles execution. Same principle, different scale.

2. Overall Description

2.1 Product perspective: The Gateway is one of four components in the Backup Management System (Gateway, Dashboard, Cloud Management Layer, Analytics Layer). It receives all external requests and forwards them to appropriate backend services.

2.2 Constraints:

  • Gateway SHALL be implemented in Go 1.22+
  • Gateway SHALL follow the project coding standards (see CS-GW-001)
  • Gateway SHALL run as a stateless container behind a load balancer

Note: constraints above are NOT FRET-formalizable — verified by inspection.
